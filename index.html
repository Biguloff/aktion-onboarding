AI-аудиоконтроль звонков для отдела продаж

Чат для методолога, который из обсуждения эталонных звонков собирает критерий-промпт и прогоняет его по корпусу транскриптов. Каждая оценка — с цитатой, баллом и предложением, что улучшить.

TL;DR

Что строим и зачем

У Актиона 2400 сотрудников и 350 тысяч компаний-подписчиков. Большой колл-центр продаёт справочные системы, продлевает подписки, обрабатывает возражения. Качество звонков — это деньги. Сейчас аудиоконтроль ручной и не масштабируется.

Один user story, одно демо

Методолог открывает чат, прогружает 2–3 эталонных звонка, итеративно обсуждает с LLM «что значит хороший звонок» (выявил потребность? отработал возражение? зафиксировал следующий шаг?). По кнопке «затеплить как критерий» LLM из истории чата формулирует промпт-критерий по строгой схеме. Методолог запускает критерий на 10–20 транскриптах — получает таблицу со скорами, цитатами и suggestion-ами. Если плохо — возвращается в чат и докручивает.

Чем отличается от готовых SaaS

Gong, Chorus, Tinkoff Speech Analytics — взрослые и зрелые, но дорогие и не дают «критерий-как-промпт через чат». Наша гипотеза: методолог хочет не предзаписанные правила, а живой диалог про правильный звонок, и из него получает рабочий критерий. Это и есть то, что отличает наш пилот от «купи Gong и не парься».

Архитектура

Как устроен MVP

Один Next.js контейнер — для UI, чата, API и фонового воркера. БД — Postgres с pgvector. LLM-провайдер абстрагирован: на демо OpenRouter, в проде — GigaChat (152-ФЗ соблюдается одним env-флагом).

Канонический контракт всей системы

Один атомарный cell — результат одного критерия по одному транскрипту. Эта форма совпадает с тем, к чему пришли Gong, Chorus, Observe.AI. Все UI, экспорты, дашборды и Kafka-события читают/пишут эту таблицу.

type EvaluationCell = {
  // identity
  cell_id:               uuid,
  tenant_id:             uuid,         // RLS-ключ (Главбух / Кадры / Юрист…)
  run_id:                uuid,         // батч-прогон
  call_id:               uuid,
  criterion_id:          uuid,
  criterion_version:     int,          // immutable lineage

  // verdict
  verdict:    'pass' | 'partial' | 'fail' | 'not_applicable',
  score:      0 | 1 | 2,                       // ordinal — RU QA convention
  confidence: 'low' | 'medium' | 'high',

  // evidence (verbatim, post-hoc verified)
  evidence: Array<{
    start_line:        int,
    end_line:          int,
    quote:             string,        // verbatim, not paraphrase
    speaker:           'agent' | 'customer' | 'unknown',
    audio_offset_ms:   int | null     // schema-ready, nullable for MVP
  }>,

  // explainability
  reasoning:   string,                   // CoT trace, hidden by default
  suggestion:  string,                   // RU coaching note

  // reproducibility
  model_id:               string,        // 'openai/gpt-5' | 'gigachat-2-max'
  criterion_prompt_hash:  string,        // regression-test PK
  transcript_hash:        string,
  temperature:            float,         // 0 для оценок
  judge_run_id:           uuid,
  created_at:             timestamp
}

План MVP

5 рабочих дней до демо

Жёсткий скоуп. Что НЕ делаем на этой неделе: реальные звонки клиентов, STT/диаризация, audio playback, полный RBAC, telephony API, SSO, k8s prod-CI/CD.

Пн
Скаффолд. Next.js 15 + Postgres + Drizzle. Миграции: users, chats, messages, transcripts, criteria, criterion_versions, evaluation_cells, runs, manager_overrides. Auth.js Credentials с email-allowlist. Залит блок shadcn.io/blocks/ai-chat-with-sidebar. Health endpoint.
Вт
Чат работает. AI SDK v5 + OpenRouter, стриминг, история сохраняется. Загрузка транскриптов (txt/json/paste). LLMClient-абстракция: OpenRouterClient рабочий, GigaChatClient — стаб с одинаковой сигнатурой. Псевдонимизатор ПДн на входе.
Ср
Кнопка «затеплить критерий». Meta-prompt, который из истории чата собирает promo-готовый промпт-критерий по схеме (verdict + evidence + suggestion). UI редактирования критерия. Версионирование. Storage в criterion_versions.
Чт
Батч-эвалюатор. BullMQ-воркер: evaluate(call, criterion) → cell. Strict JSON schema, G-Eval (reasoning перед verdict), post-hoc verbatim verification (substring match цитаты). Прогресс-бар прогона. Таблица результатов с цитатами и кнопкой «оспорить» (manager override).
Пт
Демо-готовность. Langfuse подключён. 5–7 стартовых критериев из методологического форка залиты (приветствие, SPIN-выявление потребности, отработка возражения, фиксация next step, talk-ratio). Запись 3-минутного видео-демо. Слайд «вот что есть, вот что дальше».

Риски

Топ-3 угрозы и митигации

Каждая митигация встроена в архитектуру с первого дня — не как реакция на проблему, а как защита заранее.

1 · ИБ запретит OpenRouter в день 1
Вероятность 70%
Митигация: интерфейс LLMClient с двумя реализациями (OpenRouter + GigaChat) и переключатель LLM_PROVIDER. Один env-флаг — и демо едет на корп-ключе Сбера. Запрос «корп-доступ к GigaChat 2 Max» поднимается в первый же день, параллельно с разработкой.
2 · Реальные транскрипты содержат ПДн третьих лиц
Вероятность 100%
Митигация: псевдонимизатор на входе пайплайна — regex (телефоны / email / реквизиты) + LLM-NER-проход (имена / компании). Замена на токены [ИМЯ_1], [ТЕЛ_1]. Hashmap для re-hydration хранится локально в РФ. По Приказу РКН №140 (с 01.09.2025) такая обработка — без согласия субъекта.
3 · .NET-команда отказывается принимать TS-стек
Вероятность 50%
Митигация: UI остаётся на Next.js (Action Tech уже его использует на фронте). API проектируется как REST + Kafka events с OpenAPI 3.1 спекой и snake_case Postgres-схемой (EF Core-friendly). Батч-воркер изолирован — переписывается на C# не трогая UI/API.

Вопросы команде

26 вопросов на стендап

Бери с собой и не возвращайся переспрашивать. Каждый вопрос даёт ответ, который меняет MVP в конкретной точке.

Технические — к тимлиду / .NET-лиду

  1. Где деплоим демо: локально + ngrok или dev-pod в k8s?
  2. Какая Kubernetes-инфра — managed / self-hosted, версия, есть ли Helm-стандарт?
  3. Стандарт CI/CD — GitLab CI / GitHub Actions / TeamCity?
  4. Используется ли Vault / SOPS для секретов?
  5. Есть ли prod-Postgres, в который можно проситься, или поднимать свой?
  6. Есть ли corporate Docker registry (Harbor)?
  7. Стандарт API-контрактов — OpenAPI 3.1 / gRPC / Kafka schemas (Avro/Proto)?
  8. Какие имена Kafka-топиков для звонков (action.calls.recorded, action.calls.transcribed)?
  9. Каким образом расшифровываются звонки сейчас — своя ML-команда / SaaS / нет?
  10. Стандартная аутентификация для внутренних сервисов — Keycloak / Authentik / самописный SSO?

По данным — к методологу / РОП

  1. Где сейчас хранятся записи звонков (телефония, S3, какая-то CRM)?
  2. Сколько звонков в сутки в одной команде продаж?
  3. Какая длительность среднего звонка?
  4. Дать 5–10 эталонных звонков для тестов (расшифровки или MP3).
  5. Какая текущая методика оценки звонков? Есть ли таблица критериев?
  6. Какие 3 главные боли в текущем процессе аудиоконтроля?

По комплаенсу — к ИБ / DPO / юристу

  1. Можно использовать OpenAI/OpenRouter на синтетических данных в первые 2 недели?
  2. Когда дадите корп-доступ к GigaChat 2 Max / YandexGPT 5.1 Pro?
  3. Есть ли действующий регламент обработки ПДн? Включён ли в него AI?
  4. Уведомление РКН о трансграничной передаче — оформлено?
  5. Согласия операторов на запись их голоса — собраны?
  6. Применяем ли Приказ РКН №140 об обезличивании? В каком объёме?

Продуктовые — к продактам

  1. Кто конечный пользователь MVP — методологи, РОПы, сами продавцы?
  2. Сколько ожидаем активных юзеров через 3 / 6 / 12 месяцев?
  3. KPI успеха пилота — что должно случиться, чтобы проект пошёл в прод?
  4. Есть ли выделенный бюджет на LLM-токены?
